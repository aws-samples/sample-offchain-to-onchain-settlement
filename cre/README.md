# Chainlink CRE Project

This directory contains the Chainlink Compute Runtime Environment (CRE) workflow that bridges AWS settlement instructions to on-chain execution.

## Structure

```
cre/
├── project.yaml                    # Global RPC configuration
├── secrets.yaml                    # Secret mappings (optional)
└── settlement-workflow/
    ├── main.go                     # Workflow logic (WASM)
    ├── workflow.yaml               # Workflow metadata
    ├── config.staging.json         # Staging configuration (generated by deploy script)
    └── config.production.json      # Production configuration
```

## How It Works

1. **Cron Trigger**: Workflow runs on schedule (default: every 10 seconds)
2. **Fetch Pending**: Calls `GET /instructions/pending` on AWS API
3. **Encode Report**: Converts CSI + signature into ABI-encoded report
4. **Submit On-Chain**: Sends report via KeystoneForwarder to SettlementConsumer
5. **Report Status**: Calls `POST /instructions/{id}/status` with outcome

## Configuration

### project.yaml

Defines RPC endpoints for each target environment:

```yaml
staging-settings:
  rpcs:
    - chain-name: ethereum-testnet-sepolia
      url: https://ethereum-sepolia-rpc.publicnode.com

production-settings:
  rpcs:
    - chain-name: ethereum-testnet-sepolia
      url: https://ethereum-sepolia-rpc.publicnode.com
```

### config.staging.json

```json
{
  "schedule": "*/10 * * * * *",
  "awsApiBase": "https://<api-id>.execute-api.us-east-1.amazonaws.com/prod",
  "receiverAddress": "0x<SettlementConsumerAddress>",
  "chainSelector": 16015286601757825753,
  "chainId": 11155111,
  "gasLimit": 1200000,
  "workflowRunId": "staging"
}
```

| Field | Description |
|-------|-------------|
| `schedule` | Cron expression (6-field format with seconds) |
| `awsApiBase` | AWS API Gateway base URL |
| `receiverAddress` | SettlementConsumer contract address |
| `chainSelector` | Chainlink chain selector (Sepolia: `16015286601757825753`) |
| `chainId` | EVM chain ID (Sepolia: `11155111`) |
| `gasLimit` | Gas limit for on-chain transaction |
| `workflowRunId` | Identifier for audit trail |

### Changing chains (RPC + chain selector + forwarder)

`cre workflow simulate ...` does **not** read root `.env` directly. It reads:

1. `cre/project.yaml` for RPC routing (`chain-name` + `url`)
2. `cre/settlement-workflow/config.<target>.json` for `chainSelector` and `chainId`

When switching chains, update these root `.env` values **together** before running `scripts/deploy.sh`:

- `CRE_CHAIN_NAME` (CRE EVM chain name)
- `CHAIN_SELECTOR` (Chainlink chain selector)
- `CHAIN_ID` and `RPC_URL` (EVM chain + RPC; `RPC_URL` also populates CRE `project.yaml`)
- `KEYSTONE_FORWARDER` (forwarder address for that same target network)

References:
- Chain selectors + CRE chain names: https://docs.chain.link/cre/reference/sdk/evm-client-go#chain-selectors
- KeystoneForwarder addresses by network: https://docs.chain.link/cre/guides/workflow/using-evm-client/forwarder-directory-ts#simulation-forwarders

## Running

### Simulation (Local)

```bash
cd cre
cre workflow simulate settlement-workflow --target staging-settings
```

Simulation uses a MockForwarder provided by the CRE simulator.

### Production

Deploy the workflow to Chainlink CRE infrastructure. The workflow will:
- Use the real KeystoneForwarder (Sepolia: `0xF8344CFd5c43616a4366C34E3EEE75af79a74482`)
- Submit signed reports that trigger `SettlementConsumer.onReport()`

#### Mainnet deployment checklist

Use the full **Changing chains (RPC + chain selector + forwarder)** checklist above.

Also ensure your deployed `SettlementConsumer.forwarder` constructor value matches `KEYSTONE_FORWARDER` for the target chain.

## Workflow Logic (main.go)

Key functions:

| Function | Purpose |
|----------|---------|
| `InitWorkflow` | Registers cron trigger handler |
| `onCronTrigger` | Main loop: fetch pending → encode → submit → report |
| `fetchPending` | HTTP GET to AWS API |
| `postStatus` | HTTP POST status update to AWS API |
| `toCSI` | Convert JSON instruction to Go struct |
| `encodeReport` | ABI-encode CSI + signature for on-chain consumption |

## Troubleshooting

**"awsApiBase is required"**: Update `config.staging.json` with your API Gateway URL.

**"invalid receiverAddress"**: Set `receiverAddress` to your deployed SettlementConsumer.

**"chainSelector is required"**: Ensure `chainSelector` matches your target network.

**Transaction reverts**: Check that:
- KMS signer is registered in KeyRegistry
- CSI hasn't expired
- instructionId hasn't already been executed
